# 開發執行工作流:從 SDD 到實際把專案做出來

PRD/SDD 是「規格」,這份文件定義 skill 如何**接力到實作** —— 把 SDD 的「實作路徑」變成可追蹤的 `TASK.md`,再一步步把專案開發出來。這是繼 PRD 諮詢、SDD 架構之後的第三段,讓 skill 形成「需求 → 設計 → 開發」完整閉環。

---

## 1. 觸發:生成後主動詢問(不等使用者開口)

**PRD 生成完成後**,主動提示(讓使用者知道下一步可往設計走):
```
PRD 已生成（PRD.md）。接下來要：
  [1] 進入 SDD 架構師，依 PRD 產出設計文件
  [2] 先停在 PRD，稍後再說
>
```

**SDD 生成完成後**,主動詢問是否進入實作(這是本檔的核心入口):
```
SDD 已生成（SDD.md）。要我「參考 PRD 了解需求、依 SDD 實際開發這個專案」嗎？
  [1] 是 → 規劃 TASK.md，一步步把專案開發出來
  [2] 否 → 先停在文件階段
>
```

選 `[1]` → 進入第 2 節「規劃 TASK.md」。選 `[2]` → 結束,提醒使用者日後可用主選單 `[4] 繼續/迭代現有專案` 隨時回來開發。

> 也可從主選單 `[4]` 進入:讀既有 `PRD.md`/`SDD.md`/`TASK.md`/`CHANGELOG.md` 後,若還沒有 `TASK.md` 就走第 2 節規劃;已有就走第 3 節接續開發。

---

## 2. 規劃 `TASK.md`(依 SDD 實作路徑拆任務)

開發前,先**閱讀 PRD 與 SDD**(PRD 抓需求與驗收標準、SDD 抓技術架構與「§10 實作路徑 / Implementation Roadmap」),把實作路徑的每個階段/步驟拆成可勾選的任務,寫到專案根目錄 `TASK.md`。

**任務拆解原則:**
- 一個任務 = 一個可獨立完成且可驗收的單位(對應 SDD 實作路徑的一個「步驟」或一組緊密相關步驟)。
- 依 SDD 的開發順序排列(通常:環境初始化 → 資料層 → 核心業務邏輯 → 介面 → 整合測試 → 文件),保留相依關係。
- 任務 ID 用 **`TASK-NNN`(3 位數流水號)**,對齊使用者全域 CLAUDE.md 的 `[TASK-xxx]` commit 慣例。
- 每個任務標注「對應 SDD 段落」與「驗收標準」(盡量引用 PRD 的驗收條件)。

**`TASK.md` 格式:**
```markdown
# 開發任務清單 (TASK.md)

> 由 prd-sdd-studio 依 SDD「實作路徑」規劃。commit 時對應前綴 `[TASK-NNN]`。
> 狀態圖例：⬜ 待辦 ｜ 🔵 進行中 ｜ ✅ 完成 ｜ ⏸ 阻塞

## 進度總覽
- 總任務數：N
- 已完成：X / N
- 目前進行：TASK-XXX

## 任務清單

### TASK-001 — <任務標題>
- **狀態**：⬜ 待辦
- **對應 SDD**：§10 階段 1・步驟 1.1～1.3（環境建置與專案初始化）
- **內容**：<這個任務要做什麼，條列>
- **驗收**：<怎樣算完成，盡量引用 PRD 驗收條件>
- **產出檔案**：<預計建立/修改的檔案>

### TASK-002 — ...
...
```

寫完 `TASK.md` 後,**把使用者要的任務清單先給他看過、確認**(可調整順序、合併、刪減),確認後才開工。並在 `CHANGELOG.md` 追加一筆「規劃 TASK.md」紀錄(走 workflow-local.md 流程,含主動問 git)。

---

## 3. 一步步開發(每個任務一個迴圈)

確認 `TASK.md` 後,**一次處理一個任務**,依序:

1. 取 `TASK.md` 中下一個 `⬜ 待辦` 任務,把它標成 `🔵 進行中`,更新「進度總覽」。
2. **實作這個任務**:依 SDD 對應段落的規格寫程式碼(技術棧、檔案結構、函數簽章、資料表 schema 等都遵循 SDD;遇到 SDD 沒寫清楚的細節,先問使用者再做,不要自行臆測重大決策)。
3. **驗證**:盡量實際跑起來/測試,確認符合該任務的驗收標準。有測試就跑測試。**若環境缺少完整依賴(如沒有資料庫無法啟動整個 stack),至少對「純邏輯模組」(計算、規則、驗證等不依賴 I/O 的部分)寫並實際跑通單元測試,當作「會動」的證據**,其餘依 SDD 建立腳手架並註明哪些需要外部環境才能 E2E。
4. **進入本地工作流(workflow-local.md)** —— 這是每個任務完成後的固定收尾:
   - 寫 `CHANGELOG.md`(做了什麼/為什麼/影響檔案,先標 `⏸ 未版控`)。
   - **主動問 git 同步**;同意 → 走 git 安全協議,commit message 用 **`[TASK-NNN] <繁中描述>`**(NNN = 此任務 ID)→ 回填 CHANGELOG 的 hash。
   - commit 後 → **問 PRD/SDD 文件同步跳板**(半自動,提 diff 再寫)。
5. 把 `TASK.md` 該任務標成 `✅ 完成`,更新「進度總覽」。
6. 詢問使用者要**繼續下一個任務**還是暫停:
   ```
   TASK-001 完成 ✅（X/N）。要繼續 TASK-002 嗎？
     [1] 繼續下一個
     [2] 暫停（進度已存在 TASK.md / CHANGELOG）
   >
   ```

**失敗處理(對齊全域 CLAUDE.md)**:同一個任務失敗 ≥2 次,把該任務在 `TASK.md` 標 `⏸ 阻塞`,在 CHANGELOG 記下具體錯誤、嘗試過的解法、推測 root cause,停下來告知使用者、不要硬撐。

---

## 4. 與既有文件的關係

- **PRD** = 需求與驗收來源(開發時回頭對照「有沒有滿足需求」)。
- **SDD** = 技術藍圖(怎麼蓋:技術棧、模組、schema、實作路徑)。
- **TASK.md** = 把 SDD 實作路徑落地成的可追蹤待辦,逐項勾選。
- **CHANGELOG.md** = 每個任務做了什麼的留痕。
- **文件同步跳板** = 開發過程若偏離原設計(換了套件、改了 schema),透過 workflow-local.md 的半自動流程把 PRD/SDD 更新到最新,確保「文件 ↔ 程式碼」不脫節。

> 一句話:**PRD/SDD 規劃 → TASK.md 拆解 → 逐項開發 → 每項收尾(留痕+版控+回寫文件)**,讓專案在開發中文件永遠是最新狀態。
