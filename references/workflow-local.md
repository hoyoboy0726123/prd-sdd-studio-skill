# 本地工作流:變更紀錄 + Git 協議 + 文件同步跳板

這份文件定義 skill 在「持續開發/迭代一個專案」時的本地行為。原 web app 沒有這些 —— 這是 CLI 版新增的核心價值:**每次改動都留痕、可選版控、文件永遠最新**。

---

## 1. 變更紀錄:單一 `CHANGELOG.md`(累加)

每完成**一次**對使用者 codebase 的修改/迭代,就在**專案根目錄的 `CHANGELOG.md`** 最上方(最新在上)追加一筆。檔案不存在就建立,開頭放標題:

```markdown
# 變更紀錄 (CHANGELOG)

> 由 prd-sdd-studio skill 自動維護。最新紀錄在最上方。
```

每筆紀錄格式(嚴格遵循,方便日後機器/人讀):

```markdown
## {{今日日期 YYYY-MM-DD HH:mm}} — <一句話標題>

- **做了什麼**：<具體改動,條列>
- **為什麼**：<動機 / 對應需求或 bug>
- **影響檔案**：`path/a.ts`, `path/b.py`, `PRD.md`（逐一列出實際碰到的檔案）
- **Git**：✅ 已 commit `<short-hash>` ｜ ⏸ 未版控（使用者選擇暫不 commit）
- **文件同步**：✅ PRD/SDD 已更新 ｜ ⏸ 未同步 ｜ ➖ 不適用
```

注意事項:
- **編輯 CHANGELOG 一律用結構化的檔案編輯工具(Edit/Write)逐筆追加或回填,不要用 `sed`/正則批次取代** —— 後者容易誤刪標題行或破壞 Markdown 結構。回填 commit hash 後順手檢查每筆 `## ` 標題仍完整。
- `{{今日日期}}` 一律用工具取得的**真實當下時間**,不可編造。
- 「影響檔案」要寫**實際動到的檔案**,不要寫範例佔位符。
- 一筆紀錄先以 `⏸ 未版控` 寫入;若後續使用者同意 commit,**回頭把該筆的 Git 欄位更新為 `✅ 已 commit <hash>`**。

---

## 2. 每次迭代後:主動詢問 Git 同步

完成一次修改並寫好 CHANGELOG 後,**主動**(不必等使用者開口)詢問是否同步本地 git。用 CLI 選項呈現:

```
本次修改已記錄到 CHANGELOG.md。要同步到本地 Git 版控嗎？
  [1] 是，幫我 commit
  [2] 否，暫時不要（保留為未版控）
>
```

### 選「是」→ 走 Git 安全協議(對齊使用者全域 CLAUDE.md,不可違反)

依序執行,任何一步發現異常就**停下來問**,不要硬幹:

1. **先 `git status`** —— 把這次要進版控的改動**完整列給使用者看**,讓他確認範圍。
2. **`git branch --show-current`** —— 確認目前分支。
   - **絕對禁止**直接 commit/push 到 `main`(或 `master`)。若目前在 main,先建議切到 `feat/<描述>` 分支再 commit。
   - **絕對禁止** `git push --force`(任何分支)。
   - **絕對禁止**主動 `git push`,除非使用者明確要求;本 skill 預設只做**本地 commit**。
3. **若該專案還不是 git repo**:詢問是否 `git init`(並建議建立合適的 `.gitignore`)。
4. **commit message 格式**:
   - 若此專案有對應的 Notion Task ID → 用 `[TASK-xxx] <繁中描述>`。
   - 否則 → 用描述式訊息,建議 `<type>: <繁中描述>`(type 取 feat/fix/docs/refactor/chore)。
   - 結尾依使用者全域規範附上(若該專案慣例需要)。
5. commit 完成後,**回填 CHANGELOG** 該筆的 Git 欄位為 `✅ 已 commit <short-hash>`。

### 選「否」→ 保留為 `⏸ 未版控`,不做任何 git 操作。

> ⚠️ 敏感檔案:若 `git status` 出現 `.env`、`*.key`、`*.pem`、`*credentials*`、`*secret*`、`*token*` 等(見全域 deny list),**停下來提醒使用者**,確認是否該被 `.gitignore` 忽略,不要把秘密 commit 進去。

## 3. commit 後:PRD/SDD 同步跳板(半自動)

**只有在使用者剛剛同意 commit 之後**,才接著問是否同步文件。目的:讓最初產出的 PRD/SDD 永遠反映目前的程式碼現況,當作下一輪開發的「跳板」。

流程(半自動 —— 我提 diff、使用者確認、我才寫):

```
已 commit。這次改動可能讓 PRD/SDD 與程式碼產生落差。要幫你同步文件嗎？
  [1] 是，分析並提議要改的段落
  [2] 否，這次不動文件
>
```

選「是」時:

1. **找出文件**:在專案根目錄找 `PRD.md` / `SDD.md`(或使用者指定的路徑)。都沒有就告知並跳過。
2. **分析落差**:比對「這次的程式碼改動」與現有 PRD/SDD,找出**需要更新的具體段落**(例:新增的功能要進 PRD §4 功能清單;改了資料表要更新 SDD §3.3;技術選型變動要更新 SDD §2.1)。
3. **提 diff,不直接寫**:把建議改動以「原文 → 改後」的對照呈現給使用者,**逐段或整批等他確認**。
4. **確認後才寫回**:
   - 更新對應段落內容。
   - 更新文件「文件資訊」表格的 **`最後更新` 欄位為今日日期**(`建立日期` 不動)。
   - 視情況把文件版本號 +0.1(如 1.0 → 1.1),並可在文件末尾的修訂歷史(若有)補一行。
5. **收尾**:
   - 把 CHANGELOG 該筆的「文件同步」欄位更新為 `✅ PRD/SDD 已更新`。
   - 文件本身也是改動 —— **再次主動詢問**是否要把這次的文件更新也 commit(回到第 2 節流程;commit message 建議 `docs: 同步 PRD/SDD 至最新程式碼`)。避免無限迴圈:文件同步後的 commit **不再**觸發新一輪文件同步詢問。

選「否」時:把 CHANGELOG 該筆「文件同步」欄位標為 `⏸ 未同步`。

## 4. 迭代流程總覽(串接)

```
使用者要求改動
   → 我執行修改
   → 寫 / 追加 CHANGELOG.md（先標 ⏸ 未版控、⏸ 未同步）
   → 【主動問】要 git 同步嗎？
        ├─ 是 → git status 給看 → 確認分支(非 main) → commit → 回填 CHANGELOG Git 欄位
        │        → 【接著問】要同步 PRD/SDD 嗎？
        │             ├─ 是 → 提 diff → 使用者確認 → 寫回文件 + 更新「最後更新」日期
        │             │        → 回填 CHANGELOG 文件同步欄位 → 【問】文件變更也要 commit 嗎？
        │             └─ 否 → CHANGELOG 標 ⏸ 未同步
        └─ 否 → 停在 CHANGELOG，標 ⏸ 未版控（不做 git，不問文件同步）
```

關鍵原則:
- **留痕優先**:無論使用者選不選 git,CHANGELOG 一定先寫。
- **git 在文件之前**:文件同步跳板只在「已同意 commit」之後才問。
- **半自動**:文件改動一律先給 diff、等確認,絕不無聲改寫使用者的 PRD/SDD。
- **不要無限迴圈**:文件同步產生的 commit 不再觸發下一輪文件同步。
