# 啟動偵測與續傳:每次呼叫 skill 都先做

使用者常常做一段就離開、之後再回來。所以**每次** skill 被呼叫,都要先掃描專案資料夾、判斷目前進度,**自動接力**,而不是傻傻地從頭問一遍。這份文件定義啟動時的偵測與路由邏輯。

> 核心原則:**先偵測、後問、不重來**。已經有的進度/文件一律沿用接力,絕不覆蓋或重新生成已存在的成果,除非使用者明確指示放棄。

---

## 步驟 1:確定專案工作目錄(基準 = skill 啟動的「當前目錄」)

**先用工具確認當前工作目錄**(如 `pwd`),以它為基準 —— 產物要長在使用者所在的地方,不可寫到工作區外的固定絕對路徑。

- **使用者已指明某資料夾/路徑** → 直接用它(覆寫預設)。
- **沒指明** → 判斷當前目錄本身的狀態:
  - **當前目錄(或其子資料夾)已含 PRD/SDD/TASK/CHANGELOG** → 掃描並列出這些既有專案讓使用者選續傳:
    ```
    在當前目錄偵測到以下既有專案：
      [1] kpi-system        （PRD ✅ SDD ✅ TASK ⬜ — 上次更新 <讀真實 mtime>）
      [2] <其他專案子資料夾>
      [n] ＋ 建立新專案（輸入專案名稱）
    ```
    > 「上次更新」「狀態」務必**讀各資料夾的真實檔案狀態與修改時間**填入,不可照抄本範例字串。
    - 選既有專案 → 工作目錄 = 該專案資料夾,進入步驟 2 掃描其進度。
  - **當前目錄「本身」就是一個 studio 專案**(它自己的根有 PRD/SDD/TASK)→ 別急著直接鎖定它。先看**父層**有沒有其他 studio 專案:
    - 父層還有別的專案 → 問一句:「偵測到你在 `<X>` 專案內。要續傳 `<X>`,還是回上層挑其他專案?」讓使用者能切換,不要被當前目錄綁死。
    - 父層沒有其他專案 → 直接續傳當前這個(進步驟 2)。
  - **當前目錄是空的/全新** → 直接問是否在此開新專案。
- **建立新專案** → 問專案名稱(英文 kebab-case)→ 工作目錄 = `<當前目錄>\<專案名>\`(建立);或經使用者同意直接用當前目錄當專案根。

## 步驟 2:掃描標記檔(唯讀,不修改)

在工作目錄掃描以下檔案是否存在,並讀取其內容建立脈絡:

| 檔案 | 代表意義 |
|------|----------|
| `PRD.md` | 需求文件已存在 |
| `SDD.md` | 設計文件已存在 |
| `TASK.md` | 開發任務已規劃(看勾選狀態判斷進度) |
| `CHANGELOG.md` | 過往有改動紀錄(讀最新幾筆了解上次做到哪) |
| 原始碼/專案檔(`src/`、`package.json`、`*.py` 等) | 已實際開始開發 |

也順手注意 `docs/diagrams/` 是否已有架構圖。

## 步驟 3:依掃描結果路由(決策樹)

依「找到什麼」決定如何接力,並**主動**用 CLI 選項告知使用者你偵測到的狀態:

### A. 有 `TASK.md`

先看 TASK 勾選狀態,分兩種:

#### A-1. 尚有未完成任務 → 開發到一半,**接力續開發**
1. 讀 `TASK.md`(找下一個 `⬜ 待辦` / `🔵 進行中` 任務)+ `CHANGELOG.md` 最新幾筆 + `SDD.md` 對應段落,重建脈絡。
2. 主動回報並接力:
   ```
   偵測到這個專案已有開發進度：
     - TASK.md：已完成 X / N，下一個未完成 → TASK-0XX「<標題>」
     - 最近一次變更：<CHANGELOG 最新一筆標題>
   要從 TASK-0XX 接力繼續嗎？
     [1] 是，接著開發（不重來）
     [2] 檢視整份 TASK.md / 調整任務後再繼續
     [3] 這個專案我不做了（→ 見下方「放棄舊專案」）
   ```
3. 選 `[1]` → 直接進功能 4 的「一步步開發」迴圈,從該任務做起。**不要重新規劃 TASK.md、不要重生成 PRD/SDD。**

#### A-2. TASK 全部 ✅ 完成 → 開發階段已結束,**改提供迭代/檢視/開新**
此時沒有待辦任務,不要因為「有 TASK.md」就重規劃或重生成。主動回報並給:
```
偵測到「<專案>」開發已完成（TASK N/N 全完成，最近：<CHANGELOG 最新一筆>）。要做什麼？
  [1] 迭代新功能（走本地工作流：改動 → CHANGELOG → 問 git → 文件/流程圖同步）
  [2] 檢視成果 / 跑測試 / 啟動來看
  [3] 建立新專案
```
選 `[1]` → 進功能 5 的迭代分支(改完依 workflow-local.md 收尾)。

### B. 有 `PRD.md`/`SDD.md` 但沒有 `TASK.md` → 文件已備妥,**確認後直接進開發**
這包含兩種情況:① 之前用本 skill 生成了文件還沒開發;② **使用者自己準備好的 PRD/SDD**。處理方式相同:
1. **讀文件內容**,向使用者**摘要回報你的理解**(這個專案在做什麼、核心需求、技術棧/架構重點),證明你讀懂了。
2. 詢問確認:
   ```
   偵測到專案已有現成文件：
     - PRD.md ✅（需求）   - SDD.md ✅（設計）   [若只有其一則據實說明]
   我已讀過，理解如下：<2-4 句摘要>
   要依這份 PRD/SDD 直接進入開發嗎？
     [1] 是，依現有文件規劃 TASK.md 並開始開發
     [2] 文件需要先調整/補生成（→ 進功能 1/2/3）
     [3] 重新理解需求（讓我再問你幾個問題）
   ```
3. 選 `[1]` → 進功能 4:依**現有** PRD/SDD 規劃 `TASK.md`(不重生成文件)→ 開始開發。
4. 只有 `PRD.md` 沒 `SDD.md` → 建議先進功能 3 生成 SDD;只有 `SDD.md` → 可直接規劃 TASK.md(PRD 缺漏時提醒使用者)。

### C. 只有零星原始碼、無任何文件 → 詢問如何進行
1. 回報:偵測到已有程式碼但無 PRD/SDD/TASK。
2. 提供:`[1]` 為既有程式碼補做 PRD/SDD(逆向)再開發 ｜ `[2]` 直接針對程式碼做修改/迭代(走本地工作流)｜ `[3]` 當作全新專案。

### D. 空資料夾 / 全新 → 正常顯示主選單
照 SKILL.md 主選單從功能 1 開始。

---

## 放棄舊專案 → 一定先建議「乾淨的新資料夾」

只有在使用者**明確表示**要放棄之前的開發、改做新專案時,才走這條。注意:

1. **不要刪除或覆蓋**舊專案的任何檔案 —— 舊進度留在原資料夾。
2. **強烈建議另建一個乾淨的新資料夾**再開始,避免新舊專案的 `PRD.md`/`SDD.md`/`TASK.md`/`CHANGELOG.md` 與原始碼混在一起造成偵測混亂與檔案衝突:
   ```
   了解，你要放棄目前這個專案、開始新的。
   建議「不要」在同一個資料夾重來（舊的 PRD/SDD/TASK/程式碼會混在一起）。
   我幫你在新的乾淨資料夾開始，好嗎？
     [1] 好，請建立新資料夾：<建議路徑，例 ../<新專案名>>
     [2] 我自己指定新資料夾路徑
     [3] 仍堅持用同一個資料夾（我會提醒風險並請你確認）
   ```
3. 使用者確認新資料夾後,在那裡從功能 1 全新開始。若堅持原地重來,先明確警告「舊檔會與新專案混雜」,得到二次確認才繼續。

---

## 一句話總結

**每次啟動先掃:有半成品就接力、有現成文件就讀懂後確認直接開發、要放棄就引導去乾淨新資料夾。從不無視既有進度、從不擅自重來。**
